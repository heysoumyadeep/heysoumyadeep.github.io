In the last post, we saw how databases use B-Trees to make `WHERE id = ?` fast on a billion rows. RAG systems have the exact same problem, just with a different question: **find the chunks closest in meaning to this query**.

You can't B-Tree your way through that. "Meaning" doesn't sort alphabetically. So vector databases use a different family of indexes (HNSW, IVF) built for high-dimensional similarity search. Same goal as a B-Tree: skip 99% of the data and only check what could possibly be the answer.

## Step 1: chunking

LLMs have limited context windows. You can't fit a 500-page manual into one prompt. So you split documents into smaller pieces called **chunks**, and only retrieve the few that matter for each question.

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,      # ~800 chars per chunk
    chunk_overlap=100,   # 100 chars shared with neighbors
)

chunks = splitter.split_text(document)
```

That `chunk_overlap` is important. Without it, a sentence at the boundary gets cut and the LLM loses context.

## Step 2: embeddings

An embedding turns text into a vector — a list of numbers that represents meaning as a coordinate in space. Texts with similar meaning end up near each other.

```python
from openai import OpenAI

client = OpenAI()

response = client.embeddings.create(
    model="text-embedding-3-small",
    input="how do I cancel my subscription?"
)
vector = response.data[0].embedding
# vector is a list of 1536 floats: [0.021, -0.034, ...]
```

To compare two embeddings, we measure the angle between them. This is **cosine similarity**: small angle means similar meaning.

```python
import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# 1.0 = identical meaning, 0 = unrelated, -1 = opposite
score = cosine_similarity(query_vector, chunk_vector)
```

## Step 3: storing and indexing the vectors

Now you have thousands or millions of chunk vectors. Comparing the query to every single one (called a **brute force search**) works but gets slow fast. With 1 million vectors, that's 1 million cosine calculations per query.

This is exactly the same problem B-Trees solve for SQL queries: skip the rows you don't need to check. For vectors, the most popular solution is **HNSW** (Hierarchical Navigable Small World).

The idea: build a graph where each vector is a node connected to its closest neighbors. Add multiple layers, with the top layer being a sparse "shortcut" graph and the bottom layer connecting everything densely. Search drops down through the layers, getting closer to the answer at each step.

In Postgres with pgvector, building this index is a single line:

```sql
-- Create the table
CREATE TABLE chunks (
    id BIGSERIAL PRIMARY KEY,
    content TEXT,
    embedding VECTOR(1536)
);

-- Build an HNSW index for fast similarity search
CREATE INDEX ON chunks
    USING hnsw (embedding vector_cosine_ops);
```

That second `CREATE INDEX` is doing the same job as `CREATE INDEX ON users(email)` from the database post. Same idea, different shape. Without it, every query scans every row. With it, the database walks the graph and returns the top matches in milliseconds.

In code, the runtime side looks like this:

```python
# 1. Embed the user query
query_vec = embed("how do I cancel my subscription?")

# 2. Find top-5 nearest chunks (HNSW does the work)
results = db.execute("""
    SELECT content
    FROM chunks
    ORDER BY embedding <=> %s::vector
    LIMIT 5
""", [query_vec])

# 3. Stuff retrieved chunks into the prompt
context = "\n\n".join([r["content"] for r in results])
answer = llm.complete(f"Context:\n{context}\n\nQuestion: {user_query}")
```

The `<=>` operator is pgvector's cosine distance. The `ORDER BY ... LIMIT 5` is what triggers the HNSW index. Without the index, this query scans every row.

## Trade-offs and gotchas

A few things worth knowing once you're past the basics:

**Approximate vs exact.** HNSW returns *approximate* nearest neighbors, not the exact top-k. You might miss the true best match 5% of the time. For RAG this is usually fine because retrieving "very close" chunks is good enough.

**You can't change embedding models without re-indexing.** A vector from `text-embedding-3-small` cannot be compared to one from `bge-base`. Different model = different map. Switching means re-embedding everything.

**Chunk size affects retrieval more than you'd think.** Too small (200 tokens) and you lose context. Too big (2000 tokens) and the embedding gets diluted. Start at 500-800 tokens.

**Lost in the middle.** LLMs pay more attention to the start and end of their context. If you stuff 10 chunks into a prompt, the middle ones often get ignored. Either retrieve fewer chunks or re-rank to put the most relevant first.

## What I'd actually do

1. **Start with recursive chunking, 500-800 tokens, 10-15% overlap.** Handles 80% of cases.
2. **Pick one embedding model and commit.** `text-embedding-3-small` or `bge-base` are solid defaults.
3. **Use HNSW** unless you have a specific reason not to. It's the right default in pgvector, Pinecone, Qdrant, and Weaviate.
4. **Test with 20-30 real queries** before going live. Check that the right chunk shows up in the top 5 results.
5. **Store source metadata** (filename, page) on every chunk. Show it in the answer.
6. **Re-rank for accuracy.** Pull top-20 from HNSW, then use a small re-ranker model to pick the best 5.

A RAG system is just a search engine wearing an LLM as a hat. The B-Tree was the unsung hero of relational databases. HNSW is the unsung hero of LLM apps. Same idea, new shape.
