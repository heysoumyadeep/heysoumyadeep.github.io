## The blog loading problem: two-tier architecture

The first version of the blog loaded every post upfront. With 3 posts, that was fine. With 30, the blog index would be downloading hundreds of KB of MDX content that nobody asked for.

The fix: split loading into two tiers.

**Tier 1: Metadata only (eager).** The blog index only needs titles, dates, excerpts, and tags. Not the full post body. So I load just the frontmatter for all posts upfront. This is tiny, a few KB total, and lets me render the blog listing instantly.

**Tier 2: Post body (lazy, per-post).** The full MDX component is loaded on demand only when someone opens a specific post. Vite code-splits each MDX file into its own chunk automatically.

Here's the key part of the implementation:

```js
// Tier 1: eager metadata
const metaModules = import.meta.glob('../../data/blog/posts/*.mdx', { eager: true });

// Tier 2: lazy loaders (functions that trigger dynamic imports)
const lazyLoaders = import.meta.glob('../../data/blog/posts/*.mdx');

// Build metadata index once at module init
const ALL_META = Object.entries(metaModules)
  .map(([filePath, mod]) => buildMeta(mod, filePath))
  .sort((a, b) => new Date(b.date) - new Date(a.date));

// Lazy load full post when needed
export async function getPostBySlug(slug) {
  const meta = ALL_META.find((p) => p.slug === slug);
  if (!meta) return null;

  const loaderEntry = Object.entries(lazyLoaders).find(
    ([filePath]) => slugFromPath(filePath) === slug
  );
  const mod = await loaderEntry[1]();

  return { ...meta, Component: mod.default };
}
```

The result: opening post A never downloads the content of post B. The blog index loads in under 50ms regardless of how many posts exist.

## Flash-free theming: the blocking script pattern

Dark mode is table stakes now. But most implementations flash the wrong theme on page load. You load the page, see light mode for 100ms, then it snaps to dark. Jarring.

The problem: React hydrates after the page loads. By the time your theme hook runs, the browser has already painted the default theme.

The fix: apply the theme before React even starts. I inject a tiny blocking script in `index.html` that reads `localStorage` and sets the `data-theme` attribute on `<html>` synchronously, before any CSS is evaluated.

```html
<script>
  (function() {
    const saved = localStorage.getItem('portfolio:theme');
    const theme = (saved === 'light' || saved === 'dark') ? saved : 'dark';
    document.documentElement.setAttribute('data-theme', theme);
  })();
</script>
```

This runs before the first paint. No flash. The React hook just syncs with what's already there.

Bonus: I use the View Transitions API for the theme toggle. When you click the toggle, the new theme ripples out from the button position with a circular reveal. Smooth, native, no animation library needed.

```js
const transition = document.startViewTransition(() => {
  setTheme(nextTheme);
  document.documentElement.setAttribute('data-theme', nextTheme);
});

transition.ready.then(() => {
  document.documentElement.animate(
    { clipPath: [`circle(0px at ${x}px ${y}px)`, `circle(${endRadius}px at ${x}px ${y}px)`] },
    { duration: 500, easing: 'ease-in-out', pseudoElement: '::view-transition-new(root)' }
  );
});
```

## Custom Vite plugins: SSG, sitemap, OG images

SPAs are great for interactivity, terrible for SEO. Crawlers don't always execute JavaScript. Even when they do, they don't wait for async data fetching.

The fix: static site generation. I wrote three custom Vite plugins that run at build time:

**1. SSG plugin.** Copies `dist/index.html` to every route directory (`/blog/`, `/blog/post-slug/`, etc.) so every URL serves HTML immediately. No JS execution required.

It also injects per-route Open Graph tags. Each blog post gets its own title, description, and image in the meta tags, so link previews on Twitter, LinkedIn, Slack, etc. show the correct content.

```js
function injectOgTags(html, { title, description, url, image }) {
  html = html.replace(/<title>[^<]*<\/title>/, `<title>${title}</title>`);
  html = html.replace(
    /<meta property="og:title" content="[^"]*"/,
    `<meta property="og:title" content="${title}"`
  );
  // ... same for description, url, image
  return html;
}
```

**2. Sitemap plugin.** Auto-generates `sitemap.xml` and `rss.xml` by scanning the MDX posts directory. Add a new post, rebuild, and it's in the sitemap. No manual updates.

**3. OG image plugin.** Generates a branded Open Graph image for each blog post using `@resvg/resvg-js`. Each post gets a 1200×630 PNG with the title and excerpt overlaid on a warm gradient background. These are the images that appear when you share a link on WhatsApp, LinkedIn, or Slack. Fully automated - add a post, rebuild, done.

These three plugins turn a client-side SPA into a statically-generated site with full SEO coverage. Crawlers see HTML. Link previews work. RSS readers get a feed. All without a framework like Next.js or Astro.

## Blog card preview images

The OG images are for social sharing. The blog index cards needed something different - a visual that reflects the actual content of the post, not a generic branded template.

The solution: each MDX post exports a `PreviewImage` React component alongside its frontmatter. It's just the first SVG diagram from the post, extracted into a named export:

```mdx
export function PreviewImage() {
  return (
    <svg viewBox="0 0 720 200" ...>
      {/* the illustration */}
    </svg>
  );
}
```

`PostRepository` picks it up from the eager import and stores it in the metadata bundle. `BlogIndex` renders it directly inside the card - no image files, no network requests. Because it's a React component using CSS custom properties, it responds to light/dark mode automatically, exactly as it looks inside the post itself.

Posts without a `PreviewImage` fall back to a gradient color. The pattern is opt-in and adds zero build complexity.

Every SCSS file references these tokens. Want to change the accent color? One line. Want to adjust button padding? One token. No find-and-replace across 40 components.

This also makes theming trivial. Light and dark modes are just different values for the same variables. The components don't care.

## Path aliases: clean imports

Instead of this:

```js
import { Button } from '../../../site-container/button/Button';
import { personalInfo } from '../../../data/personal';
```

I use this:

```js
import { Button } from '@components/button/Button';
import { personalInfo } from '@data';
```

Configured in `vite.config.js`:

```js
resolve: {
  alias: {
    '@': fileURLToPath(new URL('./src', import.meta.url)),
    '@components': fileURLToPath(new URL('./src/site-container', import.meta.url)),
    '@data': fileURLToPath(new URL('./src/data', import.meta.url)),
    '@features': fileURLToPath(new URL('./src/features', import.meta.url)),
  },
}
```

Cleaner, easier to refactor, no relative path hell.

## What I'd do differently

**Custom plugins are fragile.** The SSG and sitemap plugins work, but they're tightly coupled to the project structure. If I move the posts directory or change the frontmatter format, they break. A more robust solution would use a plugin API or a separate config file.

**MDX has sharp edges.** MDX 3 is powerful, but the integration with Vite and React Router required careful setup. The two-tier loading pattern works, but it's not obvious from the docs. I spent a day debugging why eager imports weren't code-splitting properly.

**Could have used a framework.** Next.js or Astro would have handled SEO, static generation, and routing out of the box. But building custom plugins taught me more about how these tools work under the hood, and I got exactly the behavior I wanted without framework constraints.

## Performance optimizations that actually mattered

After the initial build, I ran a full audit and found several performance issues hiding in plain sight. Here's what I fixed:

**Memory leak in scroll reveal.** The IntersectionObserver cleanup was inside a `requestAnimationFrame` callback. If a component unmounted before the rAF fired, the observer never disconnected. Fixed by creating the observer upfront so cleanup always runs.

**Race conditions on navigation.** Fast navigation between blog posts caused state updates on unmounted components. Added a mounted flag to guard async operations.

**Unnecessary re-renders.** Project cards re-rendered on every parent update, restarting animations. Wrapped them in `React.memo` to prevent this.

**Scroll jank.** The navbar scroll listener fired on every scroll event, causing layout thrashing. Throttled it with `requestAnimationFrame` — now it's 60fps smooth.

**Search lag.** Blog search filtered on every keystroke. Added a 300ms debounce so it only filters after you stop typing.

**Blocking scripts.** The Buy Me a Coffee widget loaded synchronously, blocking page render. Made it async.

These weren't architectural changes — just careful attention to how React actually works. The result: no memory leaks, smooth scrolling, instant search, and faster page loads.

## What worked

**Two-tier loading was the right call.** The blog index is instant, and individual posts load fast. This pattern scales to hundreds of posts without degrading performance.

**Flash-free theming is a detail that matters.** Users notice. The blocking script is 5 lines of code and eliminates a jarring visual bug.

**Static generation without a framework is viable.** Custom Vite plugins gave me full control over the build output. I got the SEO benefits of static generation without the constraints of a framework.

**Design tokens make iteration fast.** Changing the color scheme or spacing scale is a one-file edit. This paid off when I redesigned the blog layout halfway through the project.

**Performance audits catch real bugs.** The memory leak and race conditions weren't obvious during development. Running a thorough audit after launch found issues that would have caused problems at scale.

That's my take on building a portfolio that's fast, maintainable, and SEO-friendly. The patterns here (two-tier loading, flash-free theming, custom build plugins, design tokens, performance-first React) work beyond portfolios. I've used variations of them in production apps at scale. Thanks and peace.
