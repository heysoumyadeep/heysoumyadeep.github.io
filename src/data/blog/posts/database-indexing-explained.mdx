But it isn't. It's a B-Tree. Or a hash table. Or an LSM tree. Your database is quietly building one of these data structures for you, and which one it picks changes everything about how your system performs.

## What an index actually is

Think of a phone book. If you want to find "Soumyadeep," you don't read every name from page 1. The book is sorted alphabetically, so you flip to "S" and you're there in seconds.

An index is the same idea: a sorted shortcut, stored on disk, that points back to your actual rows. Without it, the database has to read every single row to find what you want. With it, it can jump straight to the answer.

```sql
-- Without an index: read all 10 million rows. Slow.
SELECT * FROM users WHERE email = 'a@b.com';

-- After this, the same query takes about 3 disk reads.
CREATE INDEX idx_users_email ON users(email);
```

But there's no free lunch. You're trading three things to get faster reads:

- **Disk space.** The index is a separate file living next to your table.
- **Write speed.** Every INSERT, UPDATE, or DELETE now has to update the index too.
- **Memory.** The database keeps hot indexes in RAM to make lookups even faster.

Hold onto these three trade-offs. They'll matter later.

## B-Trees: the default

When you write `CREATE INDEX` in Postgres, MySQL, SQL Server, or Oracle, you get a B-Tree (technically a B+Tree). It's the default because it's good at almost everything.

A B-Tree is just a sorted tree, like a tournament bracket but flipped upside down. You start at the top, ask "is my value smaller or bigger than this?", and follow the right branch down. Each node on the way down narrows your search.

```sql
SELECT * FROM orders
WHERE created_at BETWEEN '2026-01-01' AND '2026-01-31'
ORDER BY created_at;
```

### One trap to remember

This catches everyone exactly once:

```sql
-- Uses the index. Fast.
SELECT * FROM users WHERE email = 'a@b.com';

-- Skips the index entirely. Reads every row.
SELECT * FROM users WHERE LOWER(email) = 'a@b.com';
```

Why? The index is sorted by `email`, not by `LOWER(email)`. The function changes the value, and the database can't use the sorted shortcut anymore. Same thing happens if you compare a number column to a string by accident.

The fix is either an index built specifically on `LOWER(email)`, or storing the email lowercased to begin with.

## Hash indexes: fast but limited

A hash index works like a coat check at a restaurant. You hand over your ticket number, and they go straight to the matching hook. Lookup is instant, no searching.

The catch: the hooks are arranged in random order. So if you ask "give me everyone whose ticket starts with 4," they have to check every hook. Hash indexes only help with exact matches:

- `WHERE email = 'x'` works.
- `WHERE email LIKE 'x%'` doesn't.
- `WHERE email > 'x'` doesn't.
- `ORDER BY email` doesn't.

That's why B-Trees are usually preferred. They handle equality just fine *and* support ranges and sorting. Hash indexes only win in narrow cases (very large equality-only columns), which is why most of us never reach for them.

## LSM Trees: built for heavy writes

Imagine you're a librarian and books arrive every minute. With a B-Tree, you'd have to walk into the right aisle and slot each book in its correct alphabetical spot. Exhausting.

LSM trees say: "Just dump new books in a small bin near the door. Once the bin fills up, sort it and stack it. Later, merge older stacks together when you have time." Writes become fast and sequential.

This is what powers Cassandra, RocksDB, LevelDB, and ScyllaDB.

Three things happen:

1. **Writes go straight to memory** (the "Memtable", a small sorted bin in RAM).
2. **When the Memtable fills up,** it gets flushed to disk as a sorted file called an SSTable. These files are never modified, just written once and left alone.
3. **A background process called compaction** merges older SSTables together to keep things tidy.

The trade-off: writes are blazing fast, but reads might have to check several SSTables to find a value. Smart filters (Bloom filters) help skip files that definitely don't have the key.

Think back to the phone book. It's sorted by last name, then first name. So you can find "Pradhan, Soumyadeep" quickly. You can also find everyone with last name "Pradhan." But what if I asked you for everyone named "Soumyadeep" regardless of last name? You'd have to read the whole book.

Same rule for composite indexes: you can use them as long as your query starts from the leftmost column. This single rule explains most "why isn't my index working?" confusion.

A useful tip: **put equality columns first, range columns last.** `WHERE user_id = ? AND created_at > ?` works perfectly with `(user_id, created_at)`.

## When indexes hurt you

Indexes aren't free. Every index you add has to be updated on every INSERT, UPDATE, and DELETE. A table with 8 indexes does roughly **9x the work** on every write.

I've seen teams add "just one more index" until checkout latency doubled and nobody could explain why. Watch out for:

- **Unused indexes.** If nobody queries on a column, the index is just write tax. In Postgres, run `pg_stat_user_indexes` to find indexes nobody is reading.
- **Duplicate coverage.** An index on `(user_id, created_at)` already handles queries on `(user_id)` alone. The standalone one is dead weight.
- **Indexes on rarely-filtered columns.** A boolean `is_active` column has only two values, so an index on it isn't much help.

The rule I've internalized: **add an index when you have a measured slow query, not because a column "feels" indexable.**

## What I'd actually do

A short checklist before shipping anything that touches a real database:

1. **Run `EXPLAIN ANALYZE`** on every query that hits a big table. If you see `Seq Scan` and "Rows Removed by Filter" in the thousands, you've found your problem.
2. **Index your foreign keys.** Most ORMs don't do this for you, and joins on unindexed FKs are silent killers.
3. **Order composite columns deliberately:** equality first, range last.
4. **Audit unused indexes** every month or so.
5. **Never add an index "just in case."** If you can't point to a slow query it fixes, you're paying for nothing.

## In short:

An index is a tax on writes that you pay for cheaper reads. Pick the structure that matches your workload, the columns that match your queries, and the maintenance you can keep up with. Next time a 30-second query drops to 30 milliseconds, you'll know exactly which sorted structure just saved your evening.
